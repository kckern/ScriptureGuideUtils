# Cross-Canon Concordance

**Status of this document:** requirement reference (durable)
**Representative example:** LDS ↔ COC/RLDS Book of Mormon
**Last verified against code:** 2026-06-11 (v1.0.88)

## The requirement

Different religious traditions publish the *same underlying text* with
*different versification* — different chapter breaks, verse breaks, and book
ordering. A reference like "1 Nephi 3:7" therefore names **different words**
depending on whose edition you hold.

scripture-guide must be able to:

1. **Look up** a reference *within* a chosen canon's versification
   (`lookupReference('1 Nephi 1:1', { canon: 'rlds' })` must count verses the
   RLDS way, not the LDS way).
2. **Convert** verse identities *between* canons
   (`convertCanon(ids, { from: 'lds', to: 'rlds' })`), preserving the fact
   that conversions are not always 1:1 (`partial: true`).
3. **Generate** a human-readable reference in the target canon's own
   versification and naming.
4. **Round-trip** losslessly where the mapping is 1:1, and degrade explicitly
   (never silently) where it is not.

The data structure that makes #2 possible — a verse-by-verse alignment table
between two canons — is the **concordance** (stored as `_mapping.yml` per
canon).

## Why LDS ↔ COC is the representative example

The Community of Christ (formerly RLDS) and the LDS church both publish the
Book of Mormon, but from an 1879 LDS re-versification the two editions
diverged completely:

- **LDS 1 Nephi**: 22 chapters of short verses.
- **RLDS 1 Nephi**: 7 chapters of long verses (chapter 1 alone has 174 verses).

Same words, totally different coordinates. Concretely (real data from
`data/canons/rlds/_mapping.yml`):

| Identity | LDS coordinates | RLDS coordinates |
|---|---|---|
| Nephi's "I will go and do" | 1 Nephi 3:7 (verse_id 31153) | 1 Nephi 1:67–68 (verse_ids 31400–31401) |

Note the **1-to-2 split**: one LDS verse spans two RLDS verses, so
LDS→RLDS for this verse is `{ ids: [31400, 31401], partial: true }` while
RLDS→LDS for either verse is exact. This asymmetry is why the mapping format
must support multi-mappings and a `partial` flag, and why conversion APIs must
surface them.

The RLDS canon is also a **hybrid**, which exercises a second mapping kind:

- RLDS verse_ids **1–31333**: the Joseph Smith Translation (Inspired Version)
  of the Bible, mapped to the KJV `bible` canon (the JST inserts verses — e.g.
  KJV Genesis 1:1 = JST Genesis 1:3, because JST adds two prefatory verses).
- RLDS verse_ids **31334–39975**: the Book of Mormon (RLDS versification),
  mapped to the LDS canon's BOM range (LDS verse_ids 31103–37706).

So one canon (`rlds`) carries two concordance sections: `bible:` and `lds:`.

## Data model

### Verse-ID spaces

Each canon defines its own contiguous integer verse_id space in
`data/canons/<canon>/_<canon>.yml`:

```yaml
canon: rlds
id_start: 1
id_end: 39975
books:
  - key: 1_nephi
    chapters: 7
    verses: [174, 100, 255, 62, 262, 56, 68]   # verses per chapter
    first_verse_id: 31334
```

A verse_id is meaningless without knowing which canon's space it lives in.
**Cross-canon work must always carry (canon, verse_id) pairs, never bare ids.**

### Concordance format (`_mapping.yml`)

`data/canons/rlds/_mapping.yml` holds compressed alignment tables, one section
per target canon:

```yaml
lds:                       # section: maps RLDS ids -> LDS canon ids
  runs:                    # [source_start, target_start, length]
    - [31401, 31153, 5]    #   31401->31153, 31402->31154, ... (sequential 1:1)
  singles:                 # [source, target] isolated 1:1 pairs
    - [31334, 31103]
  multi:                   # [source, [targets]] one->many (sets partial: true)
    - [36448, [34626, 34627]]
```

- `runs` compress the common case (long stretches where both editions agree
  verse-for-verse) — this keeps a ~76k-line generated alignment down to
  hundreds of YAML lines.
- The loader expands sections into `forward` and `reverse` maps, so a single
  stored direction serves both `rlds→lds` and `lds→rlds`. Reverse lookups of a
  `multi` entry, and forward lookups of a target hit by several sources, also
  report `partial`.
- Mappings are **only stored on one canon** (here `rlds`); the lookup layer
  (`getMappingBetween`) checks both canons' files to resolve a direction.

### Provenance

The LDS↔RLDS BOM alignment was generated by text-similarity alignment from a
published cross-reference (see `docs/plans/2026-01-13-coc-lds-bom-mapping-design.md`;
source PDF and the 76k-line intermediate live in `_archive/`). The generator is
`build/generate-coc-mapping.cjs`.

## Required runtime behavior

With the concordance in place, the public API must satisfy:

```js
// 1. Lookup in RLDS versification (RLDS chapter/verse math, RLDS ids)
lookupReference('1 Nephi 1:67', { canon: 'rlds' })
//   -> { verse_ids: [<rlds id 31400>], ... }

// 2. Convert ids between canons
convertCanon([31153], { from: 'lds', to: 'rlds' })
//   -> { verse_ids: [31400, 31401], partial: true }

// 3. Lookup-and-convert in one call
lookupReference('1 Nephi 3:7', { convertTo: 'rlds' })
//   -> ref rendered in RLDS versification, verse_ids in RLDS space

// 4. Generate from target-canon ids
generateReference(verse_ids, { canon: 'rlds' })
//   -> '1 Nephi 1:67-68'
```

Plus invariants:

- **No silent loss**: any non-1:1 hop sets `partial: true` on the result.
- **Unmapped ranges fail loudly**: ids outside a concordance's coverage return
  an explicit error/empty result, not a wrong verse.
- **Round-trip**: `lds -> rlds -> lds` returns the original ids wherever the
  alignment is 1:1 in both directions.

## Implementation status (as of 2026-06-11)

**In place and tested:**

- Concordance data: `data/canons/rlds/_mapping.yml` with both `bible:` (JST↔KJV)
  and `lds:` (BOM↔BOM) sections, compiled to `src/data/canons/rlds/mapping.mjs`.
- Loader: `src/lib/mapping-loader.mjs` (`loadMappings`, `mapVerseId`, `mapVerse`)
  — expands runs/singles/multi, resolves direction, reports `partial`.
  Covered by the passing `test/mapping/*` suites (critical, edge-cases,
  data-integrity).
- Registry plumbing: `src/scriptcanon.mjs` has `registerCanon` /
  `registerMapping` / `convertCanon` with the `no_mapping:<from>:<to>` error
  contract.

**Not yet wired (the gap):**

- Nothing ever **calls** `registerMapping`, so at runtime
  `convertCanon([...], { from: 'lds', to: 'rlds' })` returns
  `no_mapping:lds:rlds`, and `lookupReference(..., { convertTo: 'rlds' })`
  returns empty ids. The mapping-loader and the canon registry are two halves
  that have never been connected.
- `lookupReference('1 Nephi 1:1', { canon: 'rlds' })` ignores the canon for
  versification — it parses with LDS chapter/verse math and returns LDS ids.
  RLDS book structure (`_rlds.yml`) is not used for lookup or generation.
- The pre-refactor COC adapter (which *did* convert, using `C`-prefixed string
  ids) was deliberately removed in the YAML refactor (`50d2209`, ~290 lines)
  to be replaced by this data-driven path; the unit test
  `test/unit/scriptures.test.js` › "COC canon lookup works" still encodes the
  old `C`-prefix contract and fails on master.

**Open decisions:**

1. **ID presentation**: keep the legacy `C`-prefixed string ids for RLDS/COC
   results, or present plain integers in the RLDS id space and rely on
   (canon, id) context? (The failing unit test currently assumes the former;
   the YAML data model assumes the latter.)
2. **Canon naming**: `coc` vs `rlds` — data and code were renamed to `rlds`,
   but the old public option `canon: 'coc'` still appears in tests/docs.
   Decide whether `coc` remains an accepted alias.
3. **RLDS D&C**: the RLDS canon currently covers JST + BOM only; the
   differently-numbered RLDS Doctrine & Covenants is out of scope until a
   concordance is produced for it.
